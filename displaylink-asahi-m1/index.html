Asahi M1 Max — 외장 2대는 되지만, DisplayLink는 안 쓰기로 함

Apple M1 Max 14" · stock linux-asahi 7.0 · Hyprland(Aquamarine) · 실측 기반 정리

실측 채택 안 함 Asahi DisplayLink Hyprland

핵심 한 줄: 기술적으로는 HDMI + DisplayLink로 외장 2대(듀얼 4K@60)가 된다. 실사용 CPU 부하가 너무 커서 DisplayLink는 쓰지 않기로 했다.

USB-C DisplayPort는 stock Asahi에서 atcphy_mux_set → EOPNOTSUPP로 막힌다(케이블·모니터 문제 아님). 우회인 DisplayLink(EVDI)는 화면이 나오고 배치까지 된다. 다만 DisplayLinkManager의 USB 전송 경로가 CPU를 과도하게 써서, 이 머신 실사용 구성에서는 본체 HDMI 1대(+ 내장 패널)로 남기기로 했다.

3동시 모니터 실측
4K@60×2 외장 (기술 OK)
CPU부하 과다 → 미채택
HDMI 1실사용 결론

이 글의 주제 — 세 층 판정

층질문답
1. 네이티브USB-C DP / TB / MST로 외장 2대가 되나?안 됨 — 하드웨어 파이프는 있으나 stock 드라이버가 DP mux에서 EOPNOTSUPP
2. 우회DisplayLink(EVDI)로 우회하면 화면이 나오나?됨 — 듀얼 4K@60 · 내장 포함 3모니터 · Hyprland 배치까지 실측
3. 채택그럼 일상에서 쓸 것인가?안 씀 — DisplayLinkManager CPU 부하가 실사용 기준 과다 → 본체 HDMI 1대

한 문장 주제: “외장 2대는 stock Asahi에서 네이티브로는 막히고, DisplayLink로는 열리지만, 열어도 CPU 값 때문에 닫는다.” 아래 본문은 이 세 층을 증거로 채운다.

이 머신은 “HDMI 전용 맥”이 아니다. USB-C DP용 DCP·crossbar·ATC/DPTX PHY 인프라는 stock에 있다. 막는 곳은 드라이버 한 함수다.

2. 근본 원인 — 로컬에서 재현한 실패 지점

macOS에서는 되던 DP 케이블을 같은 포트에 꽂으면, stock Asahi에서는 대략 이렇게 진행된다.

  1. 포트에 전원·USB(xHCI 허브)는 올라온다. SMC connector event까지는 간다.
  2. 포트 컨트롤러(tps6598x)가 DisplayPort alt mode를 요청한다.
  3. phy_apple_atc의 atcphy_mux_set(drivers/phy/apple/atc.c)가 EOPNOTSUPP (−95)로 실패하고 WARNING을 남긴다.
  4. DP PHY가 안 잡히니 DP-* DRM 커넥터가 생기지 않고, 모니터는 No Signal.
WARNING: drivers/phy/apple/atc.c:… at atcphy_mux_set+… [phy_apple_atc]
Workqueue: events cd321x_update_work [tps6598x]
# 레지스터에 -95 = EOPNOTSUPP (“미지원”)

같은 구성으로 여러 차례 재현됐다. 충전 전용 케이블이면 partner/alt-mode 자체가 안 뜨고, 제대로 된 DP 케이블일 때 비로소 위 WARNING이 나온다 — “요청은 되는데 드라이버가 못 받는다”.

  • USB ID: 17e9:6000 (Manufacturer: DisplayLink)
  • 듀얼 HDMI, 헤드당 3840×2160@60 실측
  • 120/144Hz 모드는 EVDI가 pixel clock 초과로 거부 (예상 범위)

리서치상 안전한 구매 축

  • 칩 명확: DL-6950 등 DL-6xxx 명시 어댑터
  • Asahi/Hyprland 커뮤니티 보고가 있는 DL 독 계열
  • 동일 모델명의 변형 주의 (DisplayLink 변형 vs alt-mode 변형)
  • 회피: SM768 / InstantView / MS2109 / FL2000 / “MST only”

4. 소프트웨어 스택 (Arch aarch64 / Asahi)

DisplayLink = 커널 EVDI DRM 장치 + 유저스페이스 DisplayLinkManager. 커널 7.x에서는 오래된 번들 EVDI로는 부족하고, Asahi 컴파일 픽스가 들어간 EVDI 1.14.16+ 가 필요하다.

구성실측 버전역할
커널linux-asahi 7.0.x (stock)호스트. 헤더 패키지로 DKMS 빌드
EVDIevdi-dkms 1.15.0가상 DRM 카드. DKMS로 현재 커널에 설치
DisplayLinkAUR displaylink 6.3 (aarch64 바이너리)USB 인코딩·전송 데몬, systemd 유닛
컴포지터Hyprland + Aquamarine멀티 카드. 필요 시 AQ_DRM_DEVICES(primary 먼저, EVDI 나중)
# 의존성 → EVDI → DisplayLink → 서비스 (개념 순서)
sudo pacman -S --needed base-devel dkms linux-asahi-headers
# AUR: evdi-dkms (또는 evdi-git) 후 displaylink
sudo systemctl enable --now displaylink.service
modprobe evdi   # 보통 서비스 ExecStartPre가 수행
lsmod | grep evdi
lsusb | grep -i 17e9
hyprctl monitors all

설치 직후 udev 스크립트가 exit 2를 몇 번 찍어도, 서비스가 재기동되며 EVDI attach·mode set까지 간 경우가 있었다. 최종 판정은 DRM 커넥터 연결과 실제 화면으로 한다.

5. 동작 실측 — 기술적으로는 된다

실패만 있는 문서가 아니다. DisplayLink 경로는 설치·출력·3모니터 배치까지 성공했다. (채택 여부는 §6.)

확인된 동작

  • 플러그 후 EVDI 카드 생성 · connector connected
  • 외장 듀얼 4K@60 (HDMI 네이티브 + DisplayLink)
  • 내장 포함 3모니터 · Hyprland 워크스페이스
  • displaylink.service enable + 재부팅 후 스택 유지

부수 함정 (채택 전 이슈)

  • 오디오 샘: 어댑터 HDMI 오디오가 default sink를 가로채 모니터로 새는 경우 → 카드 프로파일 off
  • 커넥터 이름: DVI-I-n 재부팅·재연결 시 변동 → 레이아웃 하드코딩 취약
  • 핫플러그·슬립: 커뮤니티 잔버그 보고 있음

6. 결론 — CPU 부하 때문에 쓰지 않기로 함

판정: “된다” ≠ “쓸 만하다”. DisplayLink는 이 머신에서 미채택.

DisplayLink ≠ CPU로 화면을 그린다에 가깝다. 구조는 이렇다:

  1. 컴포지터(Hyprland)가 Apple GPU로 프레임을 그린다.
  2. 본체 HDMI는 디스플레이 엔진 스캔아웃 → CPU≈0.
  3. DisplayLink는 프레임버퍼 readback → 소프트웨어 압축 → USB bulk 를 DisplayLinkManager가 CPU로 처리한다.

실측에서 이 3번 구간의 부하가 실사용 기준에서 용납하기 어려울 정도로 컸다(4K·화면 변화가 있을 때 특히). 정적 창만 DL에 두는 식의 완화는 가능하지만, 그 제약을 안고 외장 2대를 유지할 가치는 이 환경에서 없다고 판단했다.

실사용 결론: 외장은 본체 HDMI 1대만. 내장 패널 + HDMI. DisplayLink 스택은 실험·재현용으로 남겨 둘 수 있으나 일상 구성에서는 빼기로 함. 네이티브 USB-C DP가 stock에서 열리기 전까지 외장 2대는 보류.

8. 한계 · 범위

  • 이 문서는 특정 M1 Max + stock Asahi 7.0 + Hyprland 실측이다. 다른 SoC·커널·컴포지터·해상도에서는 CPU 체감이 달라질 수 있다.
  • “동작 OK / 채택 안 함”은 이 환경의 실사용 판단이다. 저해상도·정적 UI만이면 다른 결론이 날 수 있다.
  • DisplayLink 공식 전 플랫폼 보장이 아니라, EVDI + 벤더 바이너리 조합 기록이다.
  • 네이티브 USB-C DP가 stock에서 열리면 외장 2대 문제를 다시 열 수 있다.
  • 개인 식별 정보·모니터 시리얼·호스트명·내부 네트워크 주소는 포함하지 않았다.
